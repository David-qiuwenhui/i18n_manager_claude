# Proposal: I18N词条管理工具

## Why

当前项目有20+开发者参与前端开发，所有国际化（I18N）词条需要开发手动导入到统一的公服系统。现有流程缺乏 词条 Key 定义规范化和修改记录可追溯性， 所以导致词条管理混乱、修改历史难以追踪、多人协作时容易产生冲突。现在需要建立一套工程化、规范化的词条管理体系，利用代码仓库和Git版本控制实现对整个国际化修改流程的可追溯性。

## What Changes

- **新增I18N词条代码化管理**: 将词条以代码文件形式维护在Git仓库中，支持中英文对照存储
- **新增词条版本控制**: 利用Git提交历史实现词条修改的完整追溯和审查
- **新增词条验证机制**: 提供词条格式、完整性、一致性校验， 提供词条 key 值唯一性校验（确保每个词条 key 在代码仓库中唯一, 避免重复定义），并确保导入公服系统前的数据质量
- **新增公服系统集成**: 提供工具将代码仓库中的词条自动生成 Excel 文件，开发在代码仓中利用 package.json 里的 script 命令一键生成 Excel 文件，然后手动导入到公服系统
- **新增词条可视化**: 提供词条差异查看、历史记录、缺失检测等功能

## Capabilities

### New Capabilities

- `i18n-entry-storage`: I18N词条的代码化存储能力，支持以结构化文件格式（JSON）维护中英文词条键值对
- `i18n-validation`: I18N词条校验能力，检查词条格式、中英文完整性、键名规范、重复键等
- `i18n-version-control`: I18N词条版本控制能力，基于Git实现修改历史追溯、差异对比、回滚等
- `i18n-export`: I18N词条导出能力，将代码仓库中的词条转换为公服系统可导入的格式（Excel 文件）
- `i18n-sync`: I18N词条同步能力， 支持对比代码仓库与公服系统国际化词条之间的差异

### Modified Capabilities

_(无现有能力需要修改，这是一个全新的工具)_

## Impact

### 受影响的系统

- **前端项目开发流程**: 开发者需要通过代码方式管理词条，而非直接操作公服系统

### 技术依赖

- **Git**: 版本控制核心
- **Node.js/TypeScript**: CLI工具开发语言
- **JSON**: 词条存储格式

### 团队影响

- **开发者**: 学习新的词条管理流程，通过PR方式贡献词条修改
- **代码审查者**: 需要审查词条相关的代码变更
